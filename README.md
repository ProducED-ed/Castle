# Проект: портативная квест-игра "Хроники живого замка"

## АРХИТЕКТУРА

* **Сервер (CastleServer.py):** Python (Flask, SocketIO, Eventlet), запущен на Raspberry Pi. Управляет общей логикой, звуками, таймерами, связью со всеми устройствами и является главным хранителем общего состояния игры.
* **Центральный контроллер (MAIN_BOARD_V5_COM5.ino):** Arduino Mega, подключен к серверу по USB Serial. Служит мостом между сервером и контроллерами башен.
* **Контроллеры башен:** Несколько Arduino, подключенных к `MAIN_BOARD_V5_COM5` по Serial (через порты Serial1, Serial2, Serial3, mySerial).
* **Внешние устройства (ESP32):** `wolf.ino`, `train.ino`, `chest.ino`, `safe.ino`. Подключаются к серверу по Wi-Fi (HTTP POST) через API-обработчик сервера `@app.route('/api')`.
* **Веб-интерфейс - пульт (Front.html, scripts.js):** HTML/JS для управления и мониторинга квеста через браузер. Общается с сервером через SocketIO (обработчики `@socketio.on(...)`).

## СТИЛЬ КОДА И ОГРАНИЧЕНИЯ

* **Паттерн:** Придерживаться логики стейт-машины (конечного автомата).
* **Задержки (Arduino):** В коде для микроконтроллеров избегать использования блокирующей функции `delay()` в пользу неблокирующих таймеров на `millis()` (исключение: `OpenAll()` в режиме restart и `OpenDoor()` для "skip").
* **КРИТИЧНО ВАЖНО:** Функции обновления адресных светодиодов (`strip.show()`, `FastLED.show()`) отключают прерывания. **Не вызывать их в каждом цикле loop()!** Обязательно использовать троттлинг (таймер на millis), обновляя ленту не чаще, чем раз в 30-50 мс, иначе Arduino перестает принимать Serial-команды.

## 💡 КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ ПРИНЦИПЫ

1. **CastleServer.py (Сервер) Главный поток serial():** Поток `serial()` — это "сердце" квеста. Он читает Serial от Arduino, обрабатывает логику `if flag == "..."` и отправляет команды в `serial_write_queue`. Этот поток также отвечает за вызов `check_story_and_fade_up()` для восстановления громкости.
2. **Многопоточность (API и SocketIO):** Обработчики `@app.route('/api')` (для ESP32) и `@socketio.on('time')` (для пульта) выполняются в отдельных потоках.
3. **Обработка Serial (Arduino):** Внутри длительных циклов (например, анимация радуги или ожидание сервопривода) обязательно вставлять проверку `Serial.available()`, чтобы не пропускать команды 'restart' или 'skip'. Использовать функцию `MonitorSerialBuffer()` для принудительной очистки буфера при переполнении.
4. **Дублирование логики (Страховка):** Если критическая команда от башни (например, `boy_out`) может потеряться, `MAIN_BOARD` должен уметь перехватывать текстовые логи от этой башни (начинаются с `log:`) и реагировать на них как на команду.
5. **Логирование:** На каждое событие создавай логирование уровня `DEBUG`, а если я напишу, что нужно показать событие в консоли, то значит уровня `INFO`.

* **Результат:** Разнос каналов (1 и 11) устранил "глухоту" приемников ESP32, вызванную мощным сигналом передатчика Raspberry Pi на соседних частотах.

### 2. СИСТЕМНЫЕ НАСТРОЙКИ OS (RASPBERRY PI)

* **Отключена автоматизация перезагрузки сети:**
* В `crontab -e` закомментирован скрипт `wifi_rebooter.sh` (ранее выполнялся каждую минуту).
* Причина: В конфигурации с двумя интерфейсами скрипт ошибочно диагностировал отсутствие интернета и перезагружал сетевые адаптеры, вызывая разрывы связи каждые 45-55 секунд. **НЕ ВКЛЮЧАТЬ ЭТОТ СКРИПТ БЕЗ ПОЛНОЙ ПЕРЕРАБОТКИ ЛОГИКИ ПРОВЕРКИ PING.**



### 3. ИЗМЕНЕНИЯ В КОДЕ СЕРВЕРА (CastleServer.py)

* **Устранение блокировки потоков (CPU Starvation):**
* В бесконечный цикл `while True` функции `serial()` добавлена команда `eventlet.sleep(0.1)`.
* Причина: Без этой паузы поток обработки Serial-порта занимал 100% времени ядра, не давая веб-серверу (Flask) обрабатывать входящие HTTP-запросы от ESP32 (из-за чего curl и устройства "висели").


* **Авторизация для API:**
* В функцию `before_request` добавлено исключение: `if request.path.startswith('/api'): return None`.
* Причина: Устройства ESP32 не умеют проходить процедуру логина. Ранее сервер перенаправлял их POST-запросы на `/login` (код 302), из-за чего данные от игр не доходили. Теперь API открыт для локальных устройств.



### 4. ИЗМЕНЕНИЯ В ПРОШИВКАХ ESP32 (Safe, Chest, Wolf, Train)

* **Настройка статической маршрутизации:**
* Метод инициализации Wi-Fi изменен на полный формат: `WiFi.config(local_IP, gateway, subnet);`. Где `gateway` — это IP адрес Raspberry Pi (192.168.4.1), а `subnet` — 255.255.255.0.
* Причина: Без явного указания шлюза ESP32 могла принимать пакеты, но не могла отправить ответ (TCP ACK или POST-запрос) обратно на сервер.


* **Логирование при старте:**
* В `setup()` каждого устройства добавлена команда `sendLogToServer(...)` с JSON-структурой `{"log": "ESP32 [Name] is ready..."}`.
* Это позволяет диагностировать успешную загрузку и подключение устройства в логах сервера (`castle.log`).
