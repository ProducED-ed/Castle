## Проект: /root/projects/Castle

Портативная квест-игра "Хроники живого замка". Запускается на Raspberry Pi.

---

## СТРУКТУРА ФАЙЛОВ

| Файл | Роль |
|---|---|
| `CastleServer.py` | Python-сервер (Flask + SocketIO + Eventlet) на Raspberry Pi. Главный мозг: логика, звуки, таймеры, Serial к Arduino, API для ESP32 |
| `MAIN_BOARD_V5_COM5.ino` | Arduino Mega. Роутер между сервером и башнями (Serial1/2/3/mySerial). Содержит: зельеварение, центральную башню, кристаллы, библиотеку, банк, актовый зал |
| `basket3.ino` | Башня баскетбол + пещера с троллем |
| `workshop.ino` | Башня плотника (верстак, печка) |
| `owls.ino` | Башня с совами |
| `dog.ino` | Башня с собакой |
| `wolf.ino` | ESP32: внешняя игра "Волк в лесу" |
| `train.ino` | ESP32: внешняя игра "Поезд / карта" (барельеф с 4 герконами) |
| `chest.ino` | ESP32: внешняя игра "Чемоданы" |
| `safe.ino` | ESP32: внешняя игра "Сейф с животным" |
| `Front.html` + `scripts.js` | Веб-пульт управления (браузер → SocketIO → сервер) |

## АРХИТЕКТУРА СВЯЗИ

- **Сервер ↔ MAIN_BOARD:** USB Serial
- **MAIN_BOARD ↔ башни (basket3, workshop, owls, dog):** Serial1/2/3/mySerial (MAIN_BOARD — хаб)
- **ESP32 (wolf, train, chest, safe) ↔ Сервер:** Wi-Fi HTTP POST → `/api`
- **Пульт ↔ Сервер:** SocketIO

## КРИТИЧЕСКИЕ АРХИТЕКТУРНЫЕ ПРАВИЛА

### Serial / Буфер Arduino (64 байта)
- **Никогда не читать один Serial-порт из двух мест** — race condition, строки рвутся
- **Только строгое сравнение (`==`)**, никакого `indexOf` для игровых команд и никакого Левенштейна
- **Логи от башен** начинаются с `log:` — в MAIN_BOARD делать `Serial.println(buff)` и `continue`, не пропускать в проверку команд
- **Длина логов ≤ 20-25 символов** — иначе конец обрезается
- **`strip.show()` / `FastLED.show()` отключают прерывания** — не вызывать в каждом `loop()`, троттлинг ≥ 30-50 мс; критичные команды слать 3 раза с задержкой 50 мс
- **Никакого `delay()` в MAIN_BOARD** — использовать `millis()` или `smartDelay()` с вызовом `HelpTowersHandler()` внутри
- **`ClearSatelliteBuffers()`** — вызывать во время любых пауз при массовой перезагрузке башен
- **Класс `String` в быстрых циклах** вызывает heap fragmentation → фантомные рестарты. Читать Serial посимвольно внутри анимаций
- **`static` переменные внутри функций уровней** переживают рестарт — выносить в глобальную область и сбрасывать в `ready`/`restart`
- **Антидребезг:** обычный 150-300 мс, для быстрых свайпов (магическая палочка) — 30 мс

### Сброс (ready / restart)
- Давать импульс 500 мс на электромагнитные замки при сбросе
- Оборачивать эхо-логи башен: `if (command != "restart" && command != "ready") { sendLog(...); }`
- Обнулять все глобальные флаги (`isActive`, `isFastSpinning` и т.д.)
- Флаги-предохранители против двойных триггеров (напр. `if (!isDogDoorOpened)`)

### Python-сервер
- `eventlet.sleep(0.1)` в цикле `serial()` — без него CPU 100%, Flask не обрабатывает ESP32
- После критичных `serial_write_queue.put()` вызывать `process_serial_queue()` — иначе команды накапливаются и выстреливают разом
- `/api` открыт без авторизации (ESP32 не умеют логиниться)
- Автозапуск: `sudo systemctl restart castleserver.service` / логи: `journalctl -u castleserver.service -f`

### Сеть Raspberry Pi
- `wlan0` — точка доступа "Castle" для ESP32, канал 11 (2.462 ГГц)
- `wlan1` — клиент "ProducED" (интернет/Tailscale), канал 1
- **`wifi_rebooter.sh` в crontab — закомментирован**, не включать без переработки

### Tech Jump (прыжок с пульта)
- Цепочка: `ready` → очистка буфера → `start` → активация датчиков → физическая пауза → пропуск этапа
- Нельзя просто послать команду финального уровня — микроконтроллерам нужно время на инициализацию
