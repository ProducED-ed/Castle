## Проект: /root/projects/Castle

Портативная квест-игра "Хроники живого замка". Запускается на Raspberry Pi.

---

## СТРУКТУРА ФАЙЛОВ

| Файл | Роль |
|---|---|

| `MAIN_BOARD_V5_COM5.ino` | Arduino Mega. Роутер между сервером и башнями (Serial1/2/3/mySerial). Содержит: зельеварение, центральную башню, кристаллы, библиотеку, банк, актовый зал |

## АРХИТЕКТУРА СВЯЗИ

- **Сервер ↔ MAIN_BOARD:** USB Serial
- **MAIN_BOARD ↔ башни (basket3, workshop, owls, dog):** Serial1/2/3/mySerial (MAIN_BOARD — хаб)
- **ESP32 (wolf, train, chest, safe) ↔ Сервер:** Wi-Fi HTTP POST → `/api`
- **Пульт ↔ Сервер:** SocketIO

## КРИТИЧЕСКИЕ АРХИТЕКТУРНЫЕ ПРАВИЛА

### Serial / Буфер Arduino (64 байта)
- **Никогда не читать один Serial-порт из двух мест** — race condition, строки рвутся
- **Только строгое сравнение (`==`)**, никакого `indexOf` для игровых команд и никакого Левенштейна
- **Логи от башен** начинаются с `log:` — в MAIN_BOARD делать `Serial.println(buff)` и `continue`, не пропускать в проверку команд
- **Длина логов ≤ 20-25 символов** — иначе конец обрезается
- **`strip.show()` / `FastLED.show()` отключают прерывания** — не вызывать в каждом `loop()`, троттлинг ≥ 30-50 мс; критичные команды слать 3 раза с задержкой 50 мс
- **Никакого `delay()` в MAIN_BOARD** — использовать `millis()` или `smartDelay()` с вызовом `HelpTowersHandler()` внутри
- **`ClearSatelliteBuffers()`** — вызывать во время любых пауз при массовой перезагрузке башен
- **Класс `String` в быстрых циклах** вызывает heap fragmentation → фантомные рестарты. Читать Serial посимвольно внутри анимаций
- **`static` переменные внутри функций уровней** переживают рестарт — выносить в глобальную область и сбрасывать в `ready`/`restart`
- **Антидребезг:** обычный 150-300 мс, для быстрых свайпов (магическая палочка) — 30 мс

### Сброс (ready / restart)
- Давать импульс 500 мс на электромагнитные замки при сбросе
- Оборачивать эхо-логи башен: `if (command != "restart" && command != "ready") { sendLog(...); }`
- Обнулять все глобальные флаги (`isActive`, `isFastSpinning` и т.д.)
- Флаги-предохранители против двойных триггеров (напр. `if (!isDogDoorOpened)`)

### Python-сервер
- `eventlet.sleep(0.1)` в цикле `serial()` — без него CPU 100%, Flask не обрабатывает ESP32
- После критичных `serial_write_queue.put()` вызывать `process_serial_queue()` — иначе команды накапливаются и выстреливают разом
- `/api` открыт без авторизации (ESP32 не умеют логиниться)
- Автозапуск: `sudo systemctl restart castleserver.service` / логи: `journalctl -u castleserver.service -f`

### Сеть Raspberry Pi
- `wlan0` — точка доступа "Castle" для ESP32, канал 11 (2.462 ГГц)
- `wlan1` — клиент "ProducED" (интернет/Tailscale), канал 1
- **`wifi_rebooter.sh` в crontab — закомментирован**, не включать без переработки

### WiFi Auto-Connect на Старте
- **Критическая ошибка была:** `startup_wifi_routine()` вызывала `ip link down/up`, затем `wpa_cli -i wlan1 reconfigure`. **На Raspberry Pi dhcpcd управляет wpa_supplicant — без перезапуска dhcpcd это не работает.**
- **Исправлено 2026-03-31:** Заменить `wpa_cli reconfigure` на `sudo systemctl restart dhcpcd` — это то же, что делает `handle_wifi_connect()` при ручном подключении через пульт.
- **Также добавлена:**
  - Проверка флага `/home/pi/wifi_enabled` перед автоподключением (если флаг не существует, WiFi был отключен через пульт и не подключается при старте)
  - Цикл проверки подключения с логированием SSID и IP
  - Предупреждение об ошибке, если таймаут 15 сек

### Tech Jump (прыжок с пульта)
- Цепочка: `ready` → очистка буфера → `start` → активация датчиков → физическая пауза → пропуск этапа
- Нельзя просто послать команду финального уровня — микроконтроллерам нужно время на инициализацию

## ТЕХНИЧЕСКИЙ ПУЛЬТ (Tech.html)

Страница `/tech` на сервере. Содержит:
- Кнопки: **Логи сервера**, **Перезапуск**, **Тест: Баскетбол**, **Проверить аудио**
- Таблица прошивок плат с датами изменения и последней прошивки
- Терминал (зелёный фон) для вывода логов и результатов

### Кнопка "Проверить аудио"
- Запускается через SocketIO-событие `start_audio_check` (НЕ HTTP)
- Сервер читает **собственный исходный код** (`CastleServer.py` через `__file__`) и регулярными выражениями извлекает:
  - Все вызовы `play_localized_audio("story_X")` → локализованные истории
  - Все строки вида `"story_X"` в списках (`_stories_base = [...]`) → истории из списков
  - Все `pygame.mixer.Sound('file.wav')` → эффекты
- Перед сканированием из кода удаляются docstrings (`"""..."""`, `'''...'''`) и комментарии (`#...`), чтобы не подхватить примеры из документации
- Каждый файл проверяется через `wave.open()` (стандартная библиотека Python) — это ловит битые файлы (неверный заголовок, 0 фреймов, неверная частота/битность), **не загружая весь файл в память** как pygame
- Прогресс-бар в реальном времени через SocketIO (`audio_check_start`, `audio_check_progress`, `audio_check_done`)
- **Проблема-прецедент:** `story_2_c_pl.wav` физически существовал, но pygame не мог его открыть → воспроизведение падало с ошибкой. После перезаписи файла — исправилось.
- **Проблема-прецедент:** Регулярки подхватывали примеры из docstring (`play_localized_audio("story_X")`), создавая ложные "отсутствующие" файлы → исправлено очисткой кода от комментариев/docstring перед сканированием

## АУДИОСИСТЕМА

### Каналы pygame
- `channel1` — фоновая музыка (mp3 через `pygame.mixer.music`)
- `channel3` — истории и подсказки (`story_*.wav`, `hint_*.wav`)
- `effects_pool` — пул каналов для эффектов (`bottle*.wav`, `door_*.wav` и т.д.), Round Robin при занятости

### Локализация
- `LANG_SUFFIXES = {1:'ru', 2:'en', 3:'ar', 4:'fr', 5:'uk', 6:'pl'}`
- Функция `play_localized_audio("story_X")` автоматически добавляет суффикс текущего языка → `story_X_ru.wav`
- Эффекты (`play_effect`) — не локализованы, один файл для всех языков

### Приглушение фоновой музыки
- При старте story → `serial_write_queue.put('soundon')` → MAIN_BOARD → `flagSound = 0` + `Serial2.println("soundon")` (башне Тролля)
- После окончания story → `soundoff` → музыка восстанавливается
- **Эффекты (`play_effect`) НЕ блокируются во время story** — они воспроизводятся на отдельных каналах. Если кажется что эффекты "не играли" — они просто заглушались громкой story на channel3

### Критичные файлы
- Если `play_story` не может загрузить файл → `return` с ошибкой. Последующий код (напр. `play_win_salute`) всё равно выполняется — **win не блокируется отсутствием story_66_pl.wav**

## СВАЙП (магическая палочка) — MAIN_BOARD CentralTowerGameDown()

Функция в `MAIN_BOARD_V5_COM5.ino`. Два геркона: левый (пин 0 board4) и правый (пин 1 board4).

### Машина состояний
- `case 0` (IDLE): ждём первого геркона
- `case 1` (WAIT_R): левый замкнут, ждём правый → `swipe_r`
- `case 2` (WAIT_L): правый замкнут, ждём левый → `swipe_l`
- `case 3` (COOLDOWN): после засчитанного свайпа

## БАСКЕТБОЛ — ВЗАИМОДЕЙСТВИЕ MAIN_BOARD ↔ basket3.ino

### Шифр (однобуквенный протокол basket3 → MAIN_BOARD через Serial2)
| Символ | Событие |
|---|---|
| `C` | Подтверждение watchdog (Bask WD OK) |
| `I` | boy_in_game |
| `O` | boy_out_game |
| `1` | goal_1_player |
| `2` | goal_2_player |
| `3` | win (3 гола) |
| `4` | goal_1_bot |
| `5` | goal_2_bot |
| `6` | win_robot |
| `S` | start_snitch (таймаут броска) |
| `G` | lesson_goal |
| `D` | lesson_basket_done |

### Win-последовательность
1. basket3 отправляет `3` → MAIN_BOARD получает → `Serial.println("win")` + `discoBallsActive = true` + `Fireworks HIGH`
2. CastleServer получает `win` → `play_background_music("fon19.mp3")` → `play_localized_audio("story_66")` → `play_effect` × 2 → `serial_write_queue.put('play_win_salute')` → `firework` на все ESP32
3. MAIN_BOARD получает `play_win_salute` → `level = 20` → мяч останавливается

**Важно:** если `story_66_pl.wav` отсутствует, `play_story` возвращает `return`, но `play_win_salute` всё равно отправляется — win работает корректно, просто без голосовой истории победы.

## ПРАВИЛО АКТУАЛЬНОСТИ ФАЙЛОВ

### Порядок приоритетов при работе с файлом:
1. **Если пользователь прислал файл в текущем сообщении** → работать с ним
2. **Если пользователь не прислал файл** → брать последний выданный файл из `/mnt/user-data/outputs/` этого чата
3. **Если в outputs ничего нет** → брать из `/mnt/project/` (проектные файлы)

### Почему это важно:
В рамках одного чата файл проходит через несколько итераций улучшений. Каждый раз когда Claude берёт файл из `/mnt/project/` вместо последнего outputs — все предыдущие изменения чата **теряются**. Пользователь думает что получает накопленный результат, а на деле — только последнее изменение поверх оригинала.

### Как проверять перед началом работы:
```bash
ls -la /mnt/user-data/outputs/ИМЯ_ФАЙЛА
# Если есть — использовать его, а не /mnt/project/ИМЯ_ФАЙЛА
```
